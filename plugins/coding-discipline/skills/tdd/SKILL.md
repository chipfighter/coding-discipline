铁律：**没有一个先失败的测试，就不写实现代码。**
如果是自己在本轮刚写、尚未交付且能安全重做的实现，先撤掉自己的实现再从测试开始。
**绝不删除用户已有代码、别人的工作区改动或来路不明的实现。** 面对既有实现，先加 characterization / 回归测试，再做最小修改。

## 一冒出来就打住的借口
| 你会对自己说 | 现实 |
|---|---|
| 「先写实现，测试回头补」 | 回头照着实现补的测试，只证明「代码做了它做的事」，证不了「它做对了」。 |
| 「这段先留着当参考，别真删」 | 仅当它是你本轮新写且能安全撤回时才重做；用户已有代码不准删。 |
| 「太简单了，不值当写测试」 | 有稳定测试框架时优先补最小测试；不适合自动化测试时说明替代验证。 |

## 红 → 绿 → 重构
1. **红**：写一个最小的失败测试，只测一个行为，名字说清在测啥，测真实代码别测 mock（mock 只在躲不开时用）。
2. **验红（必须真跑）**：亲眼看它**失败**，而且是**因为功能还没实现**才失败——不是因为打错字 / import 错 / 语法错。
   - 一上来就过 = 你在测已经存在的行为，测试没意义，重写。
   - 是报错不是 fail = 先把错修掉，跑到它"正确地挂"为止。
3. **绿**：写**刚好让它过**的最少代码，别顺手加功能、别改别的（YAGNI）。
4. **验绿（必须真跑）**：它过 + 其它测试还全过 + 输出干净（无报错/警告）。失败时先查实现；只有能证明测试表达错了才改测试。
5. **重构**：只在全绿之后做——去重复、改名字、抽函数；全程保持绿，不加新行为。
6. 下一个行为，回到红。

## 修 bug 也走红绿
先写一个能**复现这个 bug** 的失败测试，再动手修。这条测试同时证明"修好了"和"以后不再犯"。（根因怎么定位见 systematic-debugging。）

## 不适合标准 TDD 时
文档、纯配置、生成文件、一次性迁移、探索性原型，或项目根本没有可运行的测试框架时，不要为了形式伪造测试。说明为什么不适用，选择最接近风险的验证（lint、schema 校验、dry-run、构建、手工复现），并在完成结论里标明验证边界。

## 一个小例子（加「邮箱格式校验」）
1. **红**：先写 `assert is_valid_email("a@b.com")` 为真、`assert is_valid_email("nope")` 为假——此时函数还不存在，跑，看它因「函数没定义」而挂（不是因为打错字）。
2. **绿**：写最小实现（一个正则）让这俩过，别顺手支持一堆边角情况。
3. **重构**：全绿之后再抽常量、改名；绿灯全程不断。

## 这些是设计在报警，不是测试难写
- 测试很难写 = 接口难用，简化接口。
- 什么都得 mock = 耦合太重，解耦（依赖注入）。
- setup 巨大 = 抽 helper；还复杂就是设计该简化。
