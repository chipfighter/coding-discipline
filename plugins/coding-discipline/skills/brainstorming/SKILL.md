**硬门：需要用户拍板的产品、范围或架构选择，拿到决定前不写实现。** 如果请求已经明确方向、边界和成功标准，先用几句话复述方案即可继续，不为形式要求重复批准。

## 想跳过设计时冒出来的借口
| 你会对自己说 | 现实 |
|---|---|
| 「这需求太简单，不用设计」 | 先确认没有隐藏取舍；真是清晰的小改动，几句话说明理解后即可继续。 |
| 「边做边想更快」 | 拿着没对齐的理解开干，返工更慢。先把方向对齐再动手。 |
| 「用户大概想要 X 吧」 | 别替用户拍板。一次问一个问题问清，能给选项就给选项。 |

## 流程
1. **先看上下文**：相关文件、文档、最近 commit，别凭空设计。
2. **只问真正阻塞的问题**：目的、约束、成功标准。通常把 1–3 个相关问题集中问清；能给选项就给选项。
3. **scope 不对先拆**：发现这其实是好几个独立子系统，先喊停帮着拆成子项目、定先后，别急着细化一个该拆的大坨。
4. **给 2-3 个方案带取舍**：把你**推荐**的放最前、讲清为什么，再列备选和各自代价。
5. **按复杂度呈现设计**：长设计分段讲，在真正需要选择或确认的节点停下来；短设计一次讲完。
6. **YAGNI**：用不上的功能从设计里狠狠删掉。

## 批准之后
需要拍板的方案确认后，才进入实现（见 development）。中途哪里讲不通，随时退回来重新问清。
方案里若敲定了「长期成立、光看代码看不出来」的项目决定（技术选型 / 硬边界 / 完成标准），顺手问一句要不要记进项目引导文档（Codex 用 `AGENTS.md`，Claude Code 用 `CLAUDE.md`；怎么记见 context-hygiene）。
