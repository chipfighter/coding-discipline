---
name: verify-before-done
description: 宣称"做好了/修好了/测试过了/能跑"之前用，提交、建 PR 或 MR 之前也用——任何任务都不例外。先真跑验证命令、看到输出再下结论，证据先于断言；验证范围与声明和风险相称。
---

硬规则：**成功结论必须由「最后一次相关状态变化之后」取得的、刚好足以支撑它的最低成本证据支撑。**
- 声明超过证据 = 撒谎；证据远超声明（改个错别字跑全量测试）= 浪费。两个方向都算违反。
- 证据可以复用：引用之前拿到的证据，必须说出取得时点、并指认「其后无相关变化」（没再改代码、没 pull、没动环境）。说不出或拿不准，就重跑。

## 闸门（下任何"成功/完成"结论前）
1. 这条结论靠什么证据证明？多大的证据刚好相称？
2. 手头证据是最后一次相关变化之后拿到的吗？是——引用并说明时点；不是 / 拿不准——**完整重跑**那条命令。
3. 读全部输出 + 退出码 + 数失败条数。
4. 输出对得上结论吗？对——结论连证据一起说；不对——按真实状态说。

跳过任何一步 = 撒谎，不是验证。

## 跑不了 ≠ 跳过
命令跑不起来（环境缺、依赖坏、超时）就是**验证失败**，照实报"没验证成"和被什么挡住。不许当"这步略过"悄悄绿过去，也不许把环境问题说成"功能挂了"。

## 这些都不算"验证过"
| 想拿它当"过了" | 为什么不算 |
|---|---|
| lint / 类型检查过了 | 只证明能过检查，没证明能编译、能跑、功能对。 |
| 子 agent 报告"成功了" | 那是它说的。自己去看 diff / 实际产物核对。 |
| 回归测试"跑一次就过" | 没证明它能捕获问题：用安全的隔离方式暂时禁用本轮修复 → 必须挂 → 恢复 → 过。别用回退/覆盖用户文件的方式做这一步。 |
| "测试绿了 = 需求达成" | 需求要逐条对着勾，不是测试绿就算完。 |
| 只跑了一部分 | 不等于全过。 |

## 红线
- 不准用"应该""大概""看起来对"代替证据。
- 没跑验证前不准庆祝（"好了！""搞定！""完美！"）。
- 改了要紧东西想直接 commit / push / 提 PR 或 MR 前，先过上面的闸门。
