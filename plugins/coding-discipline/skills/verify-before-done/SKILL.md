---
name: verify-before-done
description: 宣称"做好了/修好了/测试过了/能跑"之前用，提交、建 PR 或 MR 之前也用。要求先真跑验证命令、看到输出再下结论——证据先于断言。
---

宣称做完之前，先拿出这条消息里刚跑出来的证据。没在本条消息里跑过的，就不能说它过。

## 闸门（下任何"成功/完成"结论前）
1. 这条结论靠哪个命令证明？
2. 把那条命令**完整、重新**跑一遍（别用上一次的结果）。
3. 读全部输出 + 退出码 + 数失败条数。
4. 输出对得上结论吗？对——结论连证据一起说；不对——按真实状态说。
5. 这一步走完，才能开口下结论。

跳过任何一步 = 撒谎，不是验证。

## 跑不了 ≠ 跳过
命令跑不起来（环境缺、依赖坏、权限不足、超时）就是**未验证 / 被阻塞**，不是通过，也不必伪装成测试断言失败。准确说明原因、已验证到哪里和剩余风险。

## 这些都不算"验证过"
| 想拿它当"过了" | 为什么不算 |
|---|---|
| lint / 类型检查过了 | 只证明能过检查，没证明能编译、能跑、功能对。 |
| 子 agent 报告"成功了" | 那是它说的。自己去看 diff / 实际产物核对。 |
| 回归测试"跑一次就过" | 没证明测试能捕获问题。用安全的隔离方式暂时禁用本轮修复验证会挂，再恢复修复验证会过；不要回退或覆盖用户文件。 |
| "测试绿了 = 需求达成" | 需求要逐条对着勾，不是测试绿就算完。 |
| 只跑了一部分 | 不等于全过。 |

## 红线
- 不准用"应该""大概""看起来对"代替证据。
- 没跑验证前不准庆祝（"好了！""搞定！""完美！"）。
- 改了要紧东西想直接 commit / push / 提 PR 或 MR 前，先过上面的闸门。
