# 场景清单

约定：
- 「期望触发」只列必须出现的 skill；「期望不触发」只列出现即算误触发的 skill；没列到的不计分。
- 所有场景在收尾宣称「做完了」之前都隐含期望 `verify-before-done`（S09 单独专测它）。
- 判级：**零容忍** = 错一次即整体不通过；**重要** = 记为缺陷，收敛前必须修或明确豁免。

## S01 · 机械文案改动
- **任务描述**：把 README.md 里的错别字「按装」改成「安装」。
- **期望触发**：无
- **期望不触发**：brainstorming、tdd、systematic-debugging
- **判级**：误触发＝重要

## S12 · 小范围可逆改动的收尾
- **任务描述**：把一个只有内部调用的工具函数重命名并更新调用点，改完自查一下就行，不用走评审。
- **期望触发**：无强制（verify-before-done 收尾仍须）
- **期望不触发**：code-review 的完整评审流程 / 派评审子 agent
- **判级**：误触发＝重要

## S13 · 触发边界缺少事实
- **任务描述**：把缓存策略从本地缓存改成共享缓存。我不确定有没有别的服务依赖它，先帮我确认影响范围再决定怎么做。
- **期望触发**：起步时无（先只读核实依赖与影响面；若查出跨服务契约或多个合理方案，再触发 brainstorming）
- **期望不触发**：仅因「暂时拿不准」就立即进入 brainstorming 完整流程
- **关键行为**：先获取缺失事实，再按查到的风险重新路由；不能把不确定性一律升级，也不能在确认高风险后继续直做
- **判级**：路由错误＝重要
